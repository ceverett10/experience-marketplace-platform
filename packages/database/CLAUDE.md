# Database Package (`@experience-marketplace/database`)

Prisma ORM for PostgreSQL. Central data layer for the entire platform.

## Commands

```bash
npm run db:generate --workspace=@experience-marketplace/database   # Generate client
npm run db:migrate --workspace=@experience-marketplace/database    # Run migrations
```

## Connection

Singleton `PrismaClient` with pool limit 4 (Heroku Postgres essential-1).
Exported as both `prisma` and `db` from package root.

### Multi-Tenant Core

- **Site** â†’ has many Domains, Pages, Brands (1:1), SEOConfig

- **Page** â†’ belongs to Site or Microsite

## Schema Conventions

- All IDs are `cuid()` strings
- Timestamps: `createdAt` (default now), `updatedAt` (auto-update)
- Soft deletes via status fields (ARCHIVED), not actual deletion
- Composite unique keys for multi-tenant isolation: `@@unique([siteId, slug])`
- JSON fields for flexible config (homepageConfig, seoConfig, brand colors)

## Schema Change Checklist

When modifying `prisma/schema.prisma`:

1. Make your change
2. Run `npm run db:generate --workspace=@experience-marketplace/database`
3. Run `npm run typecheck` (full monorepo â€” schema changes cascade everywhere)
4. If adding a **new enum value**:
   - `JobType` â†’ also update `packages/jobs/src/types/index.ts` (JOB_TYPE_TO_QUEUE + payload)
   - `PageType` â†’ also update 17+ files that filter on page type (sitemap, content generators, admin API)
   - `SiteStatus` â†’ check `daily-content-generator.ts`, `meta-title-maintenance.ts`, `site-roadmap.ts`
5. If adding a **new model** â†’ verify no N+1 queries in consuming code
6. Create migration: `npm run db:migrate --workspace=@experience-marketplace/database`
7. Test: `npm run test --workspace=@experience-marketplace/database`

## Common Pitfalls

1. **Slug queries** must account for prefix convention (BLOG stores `blog/my-post`, CATEGORY stores `food-tours`)
2. **Site filter**: Most queries need `where: { siteId }` for tenant isolation
3. **Microsite queries**: Use `micrositeId_slug` composite key, not `siteId_slug`
4. **Pool limit**: 4 connections â€” avoid long transactions or connection leaks
5. After schema changes, run `db:generate` AND check all downstream consumers
6. **Prisma enum â†’ string literal casts**: May need `as any` â€” known friction point with strict types
7. Schema changes require `db:generate` BEFORE `typecheck` in CI â€” order matters
